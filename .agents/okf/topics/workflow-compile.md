# Why / how it works (the lifecycle, end to end)
This is the run's STATIC, pure half — authored files in, a compiled DAG out, no process spawned. An authored
TEMPLATE (`meta.json` + `nodes/<id>/{node.json,prompt.md}`) is read by `loadTemplate` (the fail-closed §8 gate):
it scans each node, `toNodeIntent` maps the authored `TemplateNode` to a runtime `NodeIntent` — rendering the
prompt's DRIVER-* contract markers through the `contract.ts` codec (`markersFromNode` → `emitMarkers`, with
`parseMarkers` the inverse) — runs the static checks, and returns a flat `WorkflowSpec` (a bag of intents). `compile`
(dag.ts) materializes each intent into a dense `NodeSpec`, `inferEdges` derives data-flow edges purely from each
node's `io.reads ⋈ io.produces` (A wrote a path B reads back), and `stagesOf` groups nodes into longest-path
topological stages (parallel lanes per level), yielding the compiled `Workflow` (`{meta, nodes, stages, edges}`) —
or a `WorkflowError`. That `Workflow` is the seam handed to the runner (see `runner`); fusion nodes are expanded
into extra intents BEFORE this compile (see `per-node-routing-and-fusion`).
