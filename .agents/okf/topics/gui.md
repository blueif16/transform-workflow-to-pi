# Why / how it works (the lifecycle, end to end)
The GUI is a static viewer: it OWNS no data, it projects what the dev middleware distills on demand. SOURCE
‚Äî `gui/vite.config.ts` registers `/__piflow/*` plugins that read the GLOBAL `~/.piflow` registry/index LIVE
(never a committed `gui/public/index.json`): `piflowRunView` calls core's `buildRunView(runDir)` to distill a
run's real `.pi/`, `piflowAgents` reads the preset catalog, `piflowRunStream` pipes `observe.watchRun`. The
canvas (`WorkflowCanvas`) calls `loadIndex()` (`/__piflow/index.json`) ‚Üí `pickCurrentRun`, then
`loadRunView` + `loadAgentCatalog`. SHAPE‚ÜíRENDER ‚Äî `toFlowGraph(view, catalog)` maps each `RunViewNode` onto a
positioned `FlowNode` (resolving `agentType`‚Üíicon), `WorkflowNode` draws the card and paints a `NodeModeStrip`
overlay per active `ViewMode` (status/model/artifacts/basis); clicking opens `NodeHud` off `rv`. COMPANION ‚Äî
`useRunStream` opens ONE `EventSource` to `/__piflow/stream/<run>`, folds SSE frames (snapshot‚Üínode-event‚Üídone)
into live `richByNode`; `liveFlowGraph` renders a running run, and `Companion` reads the shared stream plus its
own control-session channel.
