# Why / how it works (the lifecycle, end to end)
This is the run's DYNAMIC half — it takes the compiled `Workflow` (see `workflow-compile`) and drives it to
artifacts on disk. `runFromTemplate` is the join: load → `instantiateRun` (materialize `${RUN}/.pi/nodes/<id>/`)
→ compile → `runWorkflow`. `runWorkflow` walks the topological stages in order (Promise.all per stage,
concurrency-capped, HALT on first error/blocked). Each node runs through `runNode`: create the sandbox lane
(see `sandbox`) → stage `io.reads` in → write the prompt + DRIVER-* markers → `defaultPiCommand` builds the
headless `pi -p --mode json` invocation (per-node model/provider resolved here, see `per-node-routing-and-fusion`)
→ `execRunner` spawns exactly ONE pi → collect its output back to the host run dir → VERIFY by host-stat of the
declared artifacts (plus the schema/checks/return gates, see `node-action-protocol`) → `finishNode` stamps the
verdict into the run's `.pi/` tree (`run.json` + a journal entry). Nodes never call each other; they coordinate
ONLY through files on disk — the filesystem IS the contract.
