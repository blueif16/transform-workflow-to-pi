# Why / how it works (the lifecycle, end to end)
This is the run's DYNAMIC half ‚Äî it takes the compiled `Workflow` (see `workflow-compile`) and drives it to
artifacts on disk. `runFromTemplate` is the join: load ‚Üí `instantiateRun` (materialize `${RUN}/.pi/nodes/<id>/`)
‚Üí compile ‚Üí `runWorkflow`. `runWorkflow` walks the topological stages in order (Promise.all per stage,
concurrency-capped, HALT on first error/blocked). Each node runs through `runNode`: create the sandbox lane
(see `sandbox`) ‚Üí stage `io.reads` in ‚Üí write the prompt + DRIVER-* markers ‚Üí `defaultPiCommand` builds the
headless `pi -p --mode json` invocation (per-node model/provider resolved here, see `per-node-routing-and-fusion`)
‚Üí `execRunner` spawns exactly ONE pi ‚Üí collect its output back to the host run dir ‚Üí VERIFY by host-stat of the
declared artifacts (plus the schema/checks/return gates, see `node-action-protocol`) ‚Üí `finishNode` stamps the
verdict into the run's `.pi/` tree (`run.json` + a journal entry). Nodes never call each other; they coordinate
ONLY through files on disk ‚Äî the filesystem IS the contract.
