## Q1. Best use of git as a memory substrate

**The draft decision under test** (`SKILL.md:96-97, 149`): *"git = the deep, unbounded session log. Every landed edit is a `skillsys(<node>)`/`flowCommit` commit whose message IS the record (Symptom → Root → Fix → Prevention). NEVER hand-keep a changelog file — that is the #1 memory-rot; git is the generated log. git is QUERIED (`log`/`grep`/`blame`), never loaded wholesale."*

**The SOTA landscape — 2026 is the year git-native agent memory became a named category, and it converges hard on our design:**
- **Lore** (arXiv 2603.15566, [E] https://arxiv.org/html/2603.15566v1) — the closest prior art to our commit-message-is-the-record decision. It formalizes the **"Decision Shadow"** (a diff records *what* changed, discards *why* — constraints, rejected alternatives, forward context) and repurposes the **git commit message via native git-trailers** into a structured, machine-parseable decision record, "queryable via a standalone CLI" and `git log --trailer`, *"no infrastructure beyond git."* Our `skillsys(<node>)` trailer scheme (`why/lesson/rejected/verify`, `piflow-memory-v1.md §2.1`) is an independent reinvention of Lore's exact protocol — down to the `rejected alternatives` field.
- **GitOfThoughts** (arXiv 2606.14470, [E] https://arxiv.org/html/2606.14470v1) — treats reasoning as a git history: every scored thought is a **commit**, **scores recorded as git notes**, outcomes as **tags** (`success_*`/`failed_*`); retrieval is `git log --grep`, `-S`, tag filters; a **long-lived memory branch** accumulates cross-problem insights and is **merged** across agents. Directly validates "git QUERIED via log/grep, never loaded," and the git-notes-for-scores trick is a borrow candidate for our recurrence/score stamping.
- **Projectmem** (arXiv 2606.12329, [E] https://arxiv.org/html/2606.12329) — a **local-first, event-sourced, plain-text, grep-/diff-/git-versioned** memory of typed events (issue/attempt/fix/decision/note) with a **deterministic judgment layer**: *"you tried this 2 days ago — it failed."* This is our recurrence-count signal, built. Validates the "deterministic-first; the model only distills" invariant (`SKILL.md:143`).
- **CommitDistill** (arXiv 2605.18284, [E] https://arxiv.gg/abs/2605.18284) and **git-agent-memory** (https://github.com/xoniks/git-agent-memory), **structured-git-commits** (https://github.com/srdjan/structured-git-commits) — all 2026, all mine git history into typed knowledge units (Facts/Skills/Patterns), no embeddings, plain JSON/markdown, **controlled-vocabulary trailers**. `git-agent-memory` writes a `commits.md` and **git-ignores it via `.git/info/exclude`** — precisely the "optimizer-facing, keep it out of the node's tool-sweep" discipline (`SKILL.md:139`).

**When git is a LIABILITY (the honest counter):** every one of these systems pairs git with a **query/index layer** because raw `git log` does not answer "has this failure signature recurred N times?" cheaply — GitOfThoughts adds notes+tags, Projectmem adds an event log, CommitDistill adds a TF-IDF retriever with a **calibrated abstention threshold (θ=2.5)**. Git is the *substrate of record*; it is NOT the *recurrence index*. The draft already handles this correctly — it computes recurrence by `git log --grep=<signature>` **corroborated by the run trail and the lesson's own counter** (`SKILL.md:56-59`) — but the draft under-states that grepping git is O(history) and its accuracy depends entirely on a **disciplined, controlled-vocabulary signature** in the commit message (CommitDistill's whole point). Git is also a liability for anything mutable-by-nature (a "current status" line): commits are immutable, so status belongs in `memory.md`, not git — which the draft's two-surface split already gets right.

> **RECOMMENDATION — ACCEPT `SKILL.md:96-97,149` as-is; it is a well-grounded match to a category SOTA only just named (Lore, GitOfThoughts, Projectmem, CommitDistill).** One MODIFY: the draft should cite **Lore** by name as the prior art for "the commit message IS the record" and add one sentence that recurrence-grep accuracy *depends on a controlled-vocabulary signature* (CommitDistill's θ-abstention is the failure mode of an un-disciplined grep). BORROW candidate (not yet in draft): **GitOfThoughts' git-notes-for-scores** as the home for the recurrence counter, so the count lives *in git* (auditable, mergeable) rather than only in the `memory.md` frontmatter.

## Q2. Best sources to track memory on, besides/beyond git

**The draft decision under test** (`SKILL.md:98-99`): flat markdown `memory.md` as the bounded standing-state surface on top of git; the "Pointers" section (`SKILL.md:166-169`) flags "git vs a richer store (Mastra semantic-recall)" as **under active research**.

**The SOTA landscape — the substrate options, mapped to our OPTIMIZER-facing, cross-run, BOUNDED, per-node use case:**
- **Flat markdown / plain-text files** — SkillOpt's `best_skill.md`, ACE's "playbook," Claude Code's `CLAUDE.md`, Projectmem's plain-text log, `git-agent-memory`'s `commits.md`. The two agent-memory surveys note flat files win when the corpus is **small, human-auditable, and edited by the agent itself** ("From Storage to Experience," ACL 2026 Findings, [E] https://aclanthology.org/2026.findings-acl.2069.pdf; Zhang et al. survey, [E] https://bishtref.com/articles/10.1145/3748302). **Fit: perfect for us** — our corpus is capped ~40 lines/node, optimizer-written, human-reviewed.
- **Vector / semantic store** — Mem0 (arXiv 2504.19413), Mastra semantic-recall (`topK=4`), A-MEM (ChromaDB-backed). The value is **fuzzy recall over a large unbounded corpus you cannot read whole**. Our corpus is **bounded and read whole every round** → semantic search is solving a problem we designed away. **Fit: overkill.** (Mem0's own V3 moved *toward* additive plain accumulation to reduce merge cost — evidence the heavy path is not always worth it, https://deepwiki.com/mem0ai/mem0/3.3-memory-operations.)
- **Temporal knowledge graph** — Zep/Graphiti (arXiv 2501.13956, [E] https://arxiv.org/abs/2501.13956): bi-temporal edges (`t_valid`/`t_invalid`), **edge invalidation** instead of deletion, point-in-time queries, hybrid vector+BM25+graph retrieval. Cognee (https://www.cognee.ai/blog/fundamentals/how-cognee-builds-ai-memory) is explicit about the trade: **graph+vector is warranted for multi-hop reasoning over many interconnected entities; for "shallow context, markdown-style notes may suffice."** Our per-node lesson set is neither multi-hop nor entity-dense. **Fit: overkill for the store — but its bi-temporal *validity model* is a borrow (see Q3).**
- **SQLite / relational** — Letta/MemGPT archival tier, Aider's tags cache. Value = structured queries + capacity beyond context. We don't exceed context (that's the cap's job). **Fit: overkill now; the natural Stage-2 home** if `piflowctl memory find` needs a deterministic index (the draft's promotion path, `SKILL.md:170-172`).
- **Append-only event log** — Projectmem (2606.12329), GitOfThoughts. This is **exactly what git already gives us for free** (`skillsys` commits ARE the append-only event log). **Fit: we already have it — do not add a second one.**

**The cross-cutting SOTA verdict:** the choice is **corpus-size- and read-pattern-driven**, and every survey says so. Because Leg-A memory is **bounded (cap), read-whole (no retrieval problem), optimizer-written (salience is judged by a capable model, not an extractor), and per-node (small)**, the *entire* motivation for a vector/graph/SQL store evaporates. The one thing a richer store buys that markdown+git does not — a **queryable recurrence/validity index** — is better served by (a) git-notes (Q1) and (b) the OKF codegraph link the draft already rides (`SKILL.md:60-65`).

> **RECOMMENDATION — ACCEPT flat-markdown-on-git (`SKILL.md:98-99`); MODIFY the "under active research" hedge (`SKILL.md:166-169`) to RESOLVED.** The substrate question is settled by the SOTA read-pattern argument: for a bounded, read-whole, optimizer-written, per-node corpus, a vector/graph/SQL store is measurably overkill (Cognee, Mem0-V3, the two surveys all say "markdown suffices below the multi-hop/large-corpus threshold"). Do NOT adopt Mastra semantic-recall for Leg A — it targets an *unbounded conversational* corpus we do not have. Keep SQLite named only as the **Stage-2 index** for a deterministic `memory find`, not as the memory store.

## Q3. Best update rules — append vs update vs retire, delta vs consolidation, caps & eviction

**The draft decision under test** (`SKILL.md:102-109, 145`): *"ACE incremental-delta, never full-rewrite"* — **append** (new lesson, `recurrence:1`), **update** (increment recurrence, sharpen), **retire** (superseded/stale); *"Do NOT LLM-rewrite the whole file… full-rewrite consolidation causes measured context collapse (ACE 2510.04618). When the cap is exceeded, compaction is a SEPARATE, out-of-band pass that RETIRES discrete lowest-value entries (by age × non-recurrence × graduated-to-git), never a re-summarization."* Cap ~40 lines.

**The SOTA landscape — the update/eviction vocabulary is remarkably converged, and our rule sits inside the consensus:**
- **ExpeL** (arXiv 2308.10144, [E] https://arxiv.org/html/2308.10144v3) — the near-isomorphic prior art for our update rule. Its cross-trajectory insight pool uses **exactly four operations: ADD, EDIT, UPVOTE, DOWNVOTE**, each insight carrying a **vote-weight count**; frequently-upvoted insights gain prominence, downvoted ones are edited or discarded. Our (append / update-increment-recurrence / retire) is ExpeL's (ADD / EDIT+UPVOTE / DOWNVOTE) with the **recurrence counter = ExpeL's vote weight**. This is strong independent validation that "increment a per-lesson count, edit in place, never rewrite the pool" is the right shape.
- **ACE** (arXiv 2510.04618, [E] https://arxiv.org/html/2510.04618v1) — already in our v1.5 brief; the web pass confirms the mechanism the draft cites: **grow-and-refine** with **incremental delta updates + deduplication by a curator**, and the explicit claim that **full rewrites lose history and are brittle → context collapse.** The `ace-agent/ace` repo (1.2k stars, https://ace-agent.github.io/) ships it as "static prompts → dynamic playbooks that accumulate, refine, organize." Our draft's rule is a faithful application.
- **Mem0** (arXiv 2504.19413, https://arxiv.org/html/2504.19413v1) — a **two-phase extract→update** pipeline with explicit **ADD/UPDATE/DELETE** and **"newer information wins" conflict resolution**; V3 shifted to **additive accumulation via `linked_memory_ids`** to *avoid heavy merge/delete*. Validates both our delta ops AND the caution against aggressive consolidation.
- **Zep/Graphiti** (2501.13956) — the contrast worth naming: it **never deletes**, it **invalidates** (marks `t_invalid`, keeps the fact for point-in-time history). This is a *softer* retire than ours. Our git substrate already gives us this for free (a retired lesson is still in `git log`), so we can hard-remove from `memory.md` without losing history — a place our design is *cleaner* than Zep because git is the invalidation ledger.
- **A-MEM** (arXiv 2502.12110, [E] https://arxiv.org/abs/2502.12110) — the one **partial contradiction** to flag: its **"memory evolution"** step lets adding a new note **trigger LLM updates to the contextual representations of *existing* linked notes.** That is closer to a local re-write than our strict delta rule. But A-MEM has **no cap and no failure-history-hesitation concern** (it feeds a worker at runtime); its evolution optimizes *retrieval recall*, which is not our objective. So the contradiction is scoped: A-MEM's evolution is right for an unbounded runtime-injected store, wrong for our bounded optimizer-only one. It does NOT undermine our rule; it marks its boundary.
- **The eviction literature is explicit that our metric is the right axis.** Four 2025–2026 surveys — Du et al. "Rethinking Memory" (arXiv 2505.00675), Zhang et al. (10.1145/3748302), "From Storage to Experience" (ACL 2026), and the LLM-unlearning survey (10.1007/s10462-025-11376-7) — all frame eviction as a trade-off over **age × importance × recurrence/recency**, and all stress that **forgetting is beneficial, not merely a capacity hack**: uncontrolled accumulation causes drift, bias, and interference from stale/conflicting entries ("harmful memory accumulation"). Our retire metric (**age × non-recurrence × graduated-to-git**) is a direct instance — with the piflow-specific and *better-grounded* third term (graduated-to-git = the lesson's fix is now in code, so the prevention is enforced structurally and the prose is redundant).
- **Reflexion / Generative Agents (the origins):** Reflexion (arXiv 2303.11366) established verbal reflection stored as memory and re-read; Generative Agents (arXiv 2304.03442) established the **reflection *trigger*** (fire when summed recent-event importance crosses a threshold) and **importance·recency·relevance** retrieval scoring. Our MEMORIZE cadence (`SKILL.md:120-123`, per-round after SCORE→TRIAGE→FIX→GATE) is a *deterministic* reflection trigger — arguably more disciplined than Generative Agents' importance-sum heuristic, because our trigger is a confirmed-defect event, not a soft threshold.

**Where the draft is an UNHEDGED bet (be honest):** the **~40-line cap number and the exact retire metric weighting** are asserted, not measured — v1 §10.1 already lists this as open, and no SOTA source gives a magic cap for a per-node optimizer corpus (they give the *axis*, not the *constant*). This is fine as a starting default but should be labeled a **tunable, not a law.** The draft's Invariants section (`SKILL.md:145`) states the delta rule as a law (correct — SOTA-backed) but the **cap size rides along implicitly as if equally settled** (it is not).

> **RECOMMENDATION — ACCEPT the incremental-delta / append-update-retire rule (`SKILL.md:102-109,145`); it is squarely inside the ExpeL + ACE + Mem0 consensus and among our best-grounded decisions.** Two MODIFYs: (1) cite **ExpeL's ADD/EDIT/UPVOTE/DOWNVOTE** as the near-isomorphic prior art next to ACE, and note **recurrence = ExpeL's vote weight**; (2) explicitly demote the **~40-line cap and the retire-metric weights to TUNABLE defaults** (the eviction surveys settle the *axis* age×importance×recurrence, not the *constant*) — do not let them read as laws alongside the SOTA-backed delta rule. Note the **A-MEM evolution partial-contradiction** in the draft as the scoped boundary (evolution is right for unbounded runtime-injected stores, wrong for our bounded optimizer-only one), so a future reader does not mistake it for a reason to add note-rewriting.

## Q4. Assignment / scope — per-node + one system file, assigned by the four-way triage bucket

**The draft decision under test** (`SKILL.md:111-118`): a lesson lives in the **node's `memory.md`** when the fix-surface is inside one node's envelope/`owns`/`readScope` (LAPSE/SKILL/FUNCTIONALITY), and in the **system `memory.md`** when it is a cross-node hand-off/contract/wiring (ARCH/COORDINATION). The grain question (per-node vs per-vertical vs per-run; whether the session `memoryDir` hub merges in) is flagged **under active research** (`SKILL.md:117-118, 166-169`).

**The SOTA landscape — multi-agent memory scoping is a 2025–2026 research front, and it endorses per-component + a shared tier + hierarchical namespaces:**
- **G-Memory** (NeurIPS 2025, [E] https://proceedings.neurips.cc/paper_files/paper/2025/file/136a45cd9b841bf785625709a19c6508-Paper-Conference.pdf) — "Tracing Hierarchical Memory for Multi-Agent Systems": a **three-tier hierarchy** (insight / query / interaction graphs) that keeps **per-agent private memory AND shared memory at different levels**, with hierarchical namespaces for fine-grained access control and **traceability of provenance/lineage**. This is the direct academic endorsement of our **per-node + system split**: two tiers, scoped by blast radius, with provenance (our `[[okf-slice]]` link + git). The *only* thing G-Memory has that our draft does not name is the **middle "query/team" tier** — a per-vertical grain between node and system.
- **Wu & Shu survey** (2025, "Memory in LLM-based Multi-agent Systems," [E] https://doi.org/10.36227/techrxiv.176539617.79044553/v1) — systematizes exactly our question: **per-agent private memory** (autonomy, prevents leakage) vs a **controlled shared layer** (coordination, needs access control), organized by **hierarchical namespaces** (episode/task/subtask), with **provenance tagging, selective visibility, versioning** to prevent cross-contamination. Endorses per-node-private + a governed system tier; adds the caution that the shared tier needs consistency/access rules (our disjoint-write-authority rule, `SKILL.md:101`, is the mechanism).
- **SAMEP** (arXiv 2507.10562) and blackboard-architecture work (arXiv 2507.01701) — memory partitioned into **global / group / agent-specific namespaces**, blackboard as the shared tier with access controls. Reinforces that a **two-level (node + system) split is the minimum viable scoping**, and that a **group/vertical middle tier** is a known, sometimes-warranted addition.
- **LangMem / LangGraph store** (https://langchain-ai.github.io/langmem/reference/tools/) — memory keyed by a **tuple namespace** (`(domain, subdomain, user, session)`) with hierarchical scoping, and typed as **semantic / episodic / procedural**. Our Leg A is *procedural + episodic* (lessons + git history); the tuple-namespace idea is the clean implementation shape if we ever add the per-vertical middle tier (`(workflow, vertical, node)`).
- **Mastra thread/resource scope** (from our prior brief) — the `resource`-scope idea (state shared across a user's sessions) maps to the **cross-run** axis. For piflow this is the question "does the DAG remember across *unrelated* product runs?" — SOTA does this per-owner; our template-scoped `memory.md` already persists across runs *of the same template*, which is the grain we actually want.
- **The `memoryDir` session hub merge question** — the literature is unanimous that **isolation-by-default with an explicit, access-controlled shared tier** beats a merged pool (Wu & Shu; SAMEP; blackboard papers all warn of **cross-contamination**). This directly answers the draft's open question: the human-session `memoryDir` hub should **stay separate**, not merge into the optimizer's template `memory.md` — merging a human-facing conversational hub into an optimizer-facing per-node store is exactly the cross-contamination the multi-agent literature warns against (and it would violate the draft's own "distillation, never transcript" + "optimizer-facing, never injected" invariants, `SKILL.md:139-142`).

**Where the draft is well-grounded vs an unhedged bet:** the **node + system two-tier split IS SOTA-endorsed** (G-Memory, Wu & Shu, SAMEP) — ACCEPT with confidence. The **per-node grain specifically** is also endorsed (per-agent-private is the consensus default). The genuinely open sub-question the draft correctly flags is the **middle per-vertical tier**: G-Memory and the blackboard papers *have* it (their query/group tier), we currently collapse it into "system." That is a real, named gap — not a flaw, but the draft should stop calling the whole grain question "under active research" and instead name the **specific** open item (the per-vertical middle tier) as the only unresolved part.

> **RECOMMENDATION — ACCEPT the per-node + system split assigned by blast-radius bucket (`SKILL.md:111-118`); it is directly endorsed by G-Memory (NeurIPS 2025), the Wu & Shu MAS-memory survey, and SAMEP.** MODIFY the two open flags: (1) **RESOLVE "whether the session `memoryDir` hub merges in" → it stays SEPARATE** (the MAS-memory literature is unanimous that isolation-by-default + an access-controlled shared tier beats a merged pool; merging also breaks the draft's own optimizer-facing/distillation invariants); (2) **NARROW the "grain under active research" hedge** to the single genuinely-open item — the optional **per-vertical middle tier** (G-Memory's query/group tier), which we currently collapse into "system" — rather than implying the whole node-vs-system decision is unsettled. Keep the tuple-namespace shape (LangMem) in mind if that middle tier is ever added.
