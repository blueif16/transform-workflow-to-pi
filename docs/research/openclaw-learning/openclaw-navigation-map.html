OpenClaw Navigation Map

A compact reference for finding the load-bearing OpenClaw surfaces before reading implementation details.

Source scope: vendor/openclaw

One-Screen Mental Model

Layer Where To Look Why It Matters
Product and operator path README.md Names the product, install path, gateway role, channels, security model, and dev loop.
Maintainer rules AGENTS.md Defines core/plugin boundaries, commands, validation expectations, and review posture.
CLI launcher openclaw.mjs Shows how the published binary enters the source or built runtime and enforces Node support.
Core runtime src/ Holds gateway, agents, sessions, channels, plugins, provider runtime, tools, and state code.
Plugin ring extensions/ Contains bundled providers, channels, tools, and feature integrations kept outside core ownership.
Reusable packages packages/ Contains protocol, SDK, core libraries, and shared packages used by core and extensions.
Companion apps apps/, ui/ Separates app/UI surfaces from the gateway and plugin runtime.

Boundary Rules To Remember

The shortest useful rule from AGENTS.md: core stays plugin-agnostic. Plugins cross into core through documented SDK and manifest surfaces rather than importing internal core files.

The repo is a pnpm workspace with root, UI, package, and extension members declared in pnpm-workspace.yaml. That workspace shape is the first map to use before deciding where a feature or bug belongs.
