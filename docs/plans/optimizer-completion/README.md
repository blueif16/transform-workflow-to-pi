## Master sequencing — piflow optimizer completion

### The shared-file crux: `packages/core/src/optimize/driver.ts` (verified against source)

Three tasks restructure the same driver types (`CandidateEdit`, `FixGateRecord`, `FixGateResult`):
- **A2-distiller** adds an OPTIONAL `foundRoot?: string` to `CandidateEdit` (fixer return) and threads it into the manifest. Verified PURELY ADDITIVE — `CandidateEdit` has no `foundRoot` today (driver.ts:18-32); the field already exists downstream in `distill.ts:38` (`DistillContext.foundRoot`), so A2 only bridges the fixer→distiller gap.
- **A1-land** adds `liveRoot` to `FixGateRecord` + `liveRootFor(defect)→liveDir` to `FixGateStages` — additive, but changes the manifest row shape land.ts reads.
- **B1-pareto** is the STRUCTURAL one: it makes `FixGateRecord` set-valued (N candidates + a Pareto front instead of one `candidateRef`/`verdict`) and adds `pareto.ts` + vector/merge stages. This is a breaking reshape of the exact record A1 and A2 extend.

**DECISION: strictly SEQUENCE, do NOT land one coordinated mega-type-change.** Rationale: A2's edit is trivially additive and orthogonal (one optional field); folding it into B1's set-valued reshape would couple a cheap, no-signoff, learning-closing win to a large signoff-gated rock and stall it. So:
1. **A2-distiller lands FIRST** — it authors `CandidateEdit.foundRoot` + the manifest `foundRoot` row on the CURRENT single-valued shape. Small, additive, no signoff.
2. **B1-pareto rebases onto A2's shape SECOND** — B1 owns the structural set-valued reshape and MUST preserve A2's `foundRoot` (per-candidate now) and leave a clean seam for A1's `liveRoot`. B1 is the designated "type owner" for the set-valued migration; every later toucher rebases onto B1's front shape.
3. **A1-land lands THIRD (after B1)** — A1 adds `liveRoot`/`liveRootFor` onto B1's already-set-valued record and wires `land.ts` adopt to the winning candidate of the front. This is why A1 `dependsOn: [A2, B1]`.

This ordering is forced by the DAG: A2 → B1 → A1, and A1 is on the critical path ("the overlord physically acts").

### Critical path
`A2-distiller → B1-pareto → A1-land`. A2 closes the LEARNING loop (fixer root-cause → MEMORIZE distillation → cross-invocation recurrence closes). A1 closes the ACTING loop (the physical adopt/LAND step the overlord needs to mutate live files). Both A1 and A2 are cheap (medium/medium), but B1 (large, signoff) sits BETWEEN them on the shared type, making B1 the true long pole for A1. To shorten the path to "the overlord acts," B1's OQ block (Q1–Q4) must be resolved at sign-off BEFORE wave 3 starts.

### Why this wave order
- **Wave 1 = cheap wins that unblock the most and share no contested files.** A2-distiller (unblocks B1, closes learning, no signoff), A4-memory-verb (zero deps, standalone read verb), A6-fixcycle (zero deps, localized factory fn in a fn A2/B1 don't structurally reshape). These three touch DISJOINT core files (A2→driver.ts additive; A4→memory.ts/scaffold.ts; A6→optimize-fix.ts factory-only). A2 is placed in wave 1 DESPITE optimize-fix.ts overlap with A6/B1 because its driver.ts change is the gate for B1 and it is additive — land it first and everyone rebases.
- **Wave 2 = the coupled big rocks that build on A2's shape.** B1-pareto (owns the set-valued reshape; `dependsOn A2`), B3-redesign (zero code-DAG deps but touches optimize-fix.ts/optimize-loop.ts interface blocks A2 also touched, so it follows A2). Both are signoff-gated and must not interleave their optimize-fix.ts/optimize-loop.ts edits — B1 lands its loop-CLI composition first, B3 rebases the redesign subgraph onto it.
- **Wave 3 = tasks that depend on B1's set-valued shape or A1's land step.** A1-land (`dependsOn A2, B1`), B2-tier23 (`dependsOn A2, B1` — ranks the front B1 produces), B4-fixer-binding (`dependsOn A1` — a real fixer must have a land target). A1 lands before B4 within this wave.
- **Wave 4 = the closers that depend on wave-3 land/verb outputs.** A3-compact (`dependsOn A5, A1` — its `graduated` retire-trigger reads A1's land/MEMORIZE commit sig, and its codeShifted probe consumes A5's refreshed OKF `--check`), A5-understand (`dependsOn A2, B1` — must be authored LAST against the FINAL driver.ts line numbers, per its own OQ1). A5 runs before A3 within the wave (A3 depends on A5).

### Handling the shared driver.ts / types.ts insertion points concretely
- `types.ts` and `driver.ts`: A2, B1, B2 all add sibling fields/types here. No LOGICAL conflict (all additive or B1-owned reshape) but physical insertion collisions. Coordinate by insertion ORDER = wave order: A2's `foundRoot` first, B1's front types second, B2's judgment fields third, A1's `liveRoot`/`liveRootFor` onto B1's shape. Each task rebases onto the prior; the strict wave sequencing IS the merge protocol.
- `packages/cli/src/optimize-fix.ts` is the busiest CLI file (A2, A6, B1, B2, B3, A1 all touch it). Localize each edit to its own function: A6 → `makeFixGateRunner` counter block; A2 → distill wiring after MEMORIZE; B1 → stage composition; B2 → `scoreTriageEnrich` judge insertion; A1 → adopt path. Wave ordering keeps these from racing.
- `packages/cli/src/cli.ts` HELP region: A2/A3/A4/B1/B3 all append optimize HELP text. Low-conflict text block; append-only, last writer in each wave reconciles.

### One-line summary of the DAG
A2 (learning seam, additive) unblocks B1 (set-valued reshape, the type owner) which unblocks A1 (physical land) and B2 (rank the front); A1 unblocks B4 (real fixer) and A3 (retire on land-sig); A5 refreshes anchors last; A4/A6/B3 ride mostly independent lanes gated only by shared-file insertion order.
