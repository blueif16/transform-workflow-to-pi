The Smallest Useful Model

Treat OpenClaw as a local-first assistant gateway with a narrow core and a plugin ring. The README tells you what the product does. The root AGENTS.md tells you what boundaries maintainers protect when changing it.

The first map is not a call graph. It is a boundary map: openclaw.mjs starts the CLI, src/ holds the core runtime, extensions/ holds plugin-owned capability, packages/ holds shared libraries and SDK surfaces, and apps/ plus ui/ hold companion surfaces.

Read In This Order

  1. Open vendor/openclaw/README.md and identify the operator path: install, onboard, gateway, channel, agent.
  2. Open vendor/openclaw/AGENTS.md and find the architecture rule that says core stays plugin-agnostic.
  3. Open vendor/openclaw/openclaw.mjs and note that the binary wrapper checks Node support before entering runtime code.
  4. Open vendor/openclaw/pnpm-workspace.yaml and name the workspace members: root, ui, packages/*, and extensions/*.

Why This Matters

OpenClaw has many channels, providers, apps, and tools. Without the boundary map, every question looks like a search problem. With it, you can decide whether a question belongs to operator docs, core runtime, plugin-owned capability, shared SDK/package surface, or an app/UI shell before opening deep files.
